# Чеклист ручного тестування

Цей документ містить повний перелік тестових сценаріїв для всіх нових модулів системи. Перед початком тестування переконайтеся, що виконані всі передумови для кожного розділу.

---

## 1. ПРРО Фіскалізація (Checkbox)

### Передумови
- Зареєстрований касовий апарат у ДПС
- Отримано ліцензійний ключ Checkbox
- Налаштовано підключення до Checkbox API
- Призначено касира з правами доступу
- Є тестові дані: товари з ціною та ПДВ

### 1.1. Відкриття зміни
- [ ] **Тест:** Відкрити зміну через інтерфейс
  - **Очікуваний результат:** Зміна успішно відкрита, отримано ID зміни
  - **Тестові дані:** Касир: `cashier@example.com`, Каса: `4000123456`
  - **Помилки для тестування:**
    - Спроба відкрити зміну, коли попередня не закрита
    - Відкриття зміни без ліцензії
    - Відкриття зміни без інтернет-з'єднання
  - **Перевірити:** Відображення статусу зміни в інтерфейсі

### 1.2. Створення чеку продажу
- [ ] **Тест:** Створити чек продажу з 1 товаром (готівка)
  - **Очікуваний результат:** Чек успішно створено, отримано фіскальний номер та QR-код
  - **Тестові дані:** Товар: `iPhone 15 Pro`, Ціна: `45000 грн`, Оплата: `Готівка`

- [ ] **Тест:** Створити чек продажу з кількома товарами (картка)
  - **Очікуваний результат:** Всі товари у чеку, правильна сума ПДВ
  - **Тестові дані:** 3 товари, загальна сума `12500 грн`, Оплата: `Картка`

- [ ] **Тест:** Створити чек зі знижкою
  - **Очікуваний результат:** Знижка правильно застосована, ПДВ перераховано
  - **Тестові дані:** Товар `5000 грн`, Знижка `10%`, Сума до сплати: `4500 грн`

- [ ] **Тест:** Змішана оплата (готівка + картка)
  - **Очікуваний результат:** Обидва типи оплати зафіксовані в чеку
  - **Тестові дані:** Сума `1000 грн`, Готівка: `300 грн`, Картка: `700 грн`

- [ ] **Помилки для тестування:**
  - Створення чеку з порожнім кошиком
  - Створення чеку на суму 0 грн
  - Втрата з'єднання під час фіскалізації
  - Створення чеку при закритій зміні
  - Товар без ціни або ПДВ

- [ ] **Продуктивність:** Створення чеку < 3 секунд

### 1.3. Створення чеку повернення
- [ ] **Тест:** Повернення товару за фіскальним номером
  - **Очікуваний результат:** Чек повернення створено, сума повернута
  - **Тестові дані:** Фіскальний номер попереднього чеку

- [ ] **Тест:** Часткове повернення (1 товар з кількох)
  - **Очікуваний результат:** Створено чек повернення тільки для 1 товару
  - **Тестові дані:** Чек з 3 товарами, повернення 1 товару

- [ ] **Тест:** Повернення зі знижкою
  - **Очікуваний результат:** Сума повернення = сума після знижки
  - **Тестові дані:** Товар `4500 грн` (після знижки 10%)

- [ ] **Помилки для тестування:**
  - Повернення неіснуючого чеку
  - Повернення чеку старше 14 днів (згідно політики магазину)
  - Повернення більшої кількості, ніж було продано
  - Повторне повернення того самого чеку

### 1.4. Службове внесення готівки
- [ ] **Тест:** Внесення готівки в касу
  - **Очікуваний результат:** Операція зафіксована, баланс каси оновлено
  - **Тестові дані:** Сума: `10000 грн`, Причина: `Інкасація`

- [ ] **Помилки для тестування:**
  - Внесення від'ємної суми
  - Внесення при закритій зміні

### 1.5. Службове винесення готівки
- [ ] **Тест:** Винесення готівки з каси
  - **Очікуваний результат:** Операція зафіксована, баланс каси зменшено
  - **Тестові дані:** Сума: `5000 грн`, Причина: `Здача в банк`

- [ ] **Помилки для тестування:**
  - Винесення більше, ніж є в касі
  - Винесення від'ємної суми

### 1.6. Закриття зміни (Z-звіт)
- [ ] **Тест:** Закрити зміну з кількома чеками
  - **Очікуваний результат:** Z-звіт створено, сума відповідає всім операціям
  - **Тестові дані:** 5 чеків продажу, 1 повернення, 1 внесення, 1 винесення

- [ ] **Тест:** Перевірка структури Z-звіту
  - **Очікуваний результат:** Відображаються всі розділи: продажі, повернення, готівка, картка, ПДВ

- [ ] **Тест:** Закриття зміни без операцій
  - **Очікуваний результат:** Z-звіт з нульовими сумами

- [ ] **Помилки для тестування:**
  - Закриття вже закритої зміни
  - Розбіжність між фактичною та розрахунковою сумою в касі

### 1.7. Перевірка QR-коду чеку
- [ ] **Тест:** Сканування QR-коду з чеку
  - **Очікуваний результат:** Відображається детальна інформація про чек
  - **Тестові дані:** QR-код з будь-якого чеку

- [ ] **Тест:** Перевірка чеку на сайті ДПС
  - **Очікуваний результат:** Чек знайдено в базі ДПС
  - **Тестові дані:** Фіскальний номер чеку

### 1.8. Пошук чеку за фіскальним номером
- [ ] **Тест:** Пошук існуючого чеку
  - **Очікуваний результат:** Чек знайдено, відображена вся інформація
  - **Тестові дані:** Фіскальний номер: `4000123456789`

- [ ] **Тест:** Пошук неіснуючого чеку
  - **Очікуваний результат:** Повідомлення "Чек не знайдено"

- [ ] **Тест:** Пошук за датою та сумою
  - **Очікуваний результат:** Список чеків за критеріями
  - **Тестові дані:** Дата: `2025-12-14`, Сума: `від 1000 до 5000 грн`

## 2. B2B Портал

### Передумови
- Налаштовані рівні цін (роздріб, малий опт, великий опт)
- Створені товари з різними цінами
- Налаштовані кредитні ліміти
- Підготовлені тестові файли (CSV, Excel, XML)

### 2.1. Реєстрація B2B клієнта
- [ ] **Тест:** Реєстрація нового B2B клієнта
  - **Очікуваний результат:** Клієнт зареєстрований, отримано лист з підтвердженням
  - **Тестові дані:**
    ```
    Назва компанії: ТОВ "Тест Трейд"
    ЄДРПОУ: 12345678
    Email: b2b@test.com
    Телефон: +380501234567
    ```

- [ ] **Тест:** Завантаження документів (Свідоцтво ПДВ, Статут)
  - **Очікуваний результат:** Документи завантажені, статус "На модерації"
  - **Тестові дані:** PDF-файли розміром < 5 МБ

- [ ] **Помилки для тестування:**
  - Реєстрація з існуючим ЄДРПОУ
  - Реєстрація без обов'язкових полів
  - Завантаження файлів неправильного формату (не PDF)
  - Завантаження файлів > 10 МБ

### 2.2. Перевірка цін для різних рівнів
- [ ] **Тест:** Роздрібна ціна для нового клієнта
  - **Очікуваний результат:** Відображається роздрібна ціна
  - **Тестові дані:** Товар: `Ноутбук`, Роздріб: `25000 грн`

- [ ] **Тест:** Ціна малого опту (від 10 одиниць)
  - **Очікуваний результат:** Ціна знижена на 5%
  - **Тестові дані:** Товар: `Мишка`, Роздріб: `500 грн`, Малий опт: `475 грн`

- [ ] **Тест:** Ціна великого опту (від 100 одиниць)
  - **Очікуваний результат:** Ціна знижена на 15%
  - **Тестові дані:** Товар: `Клавіатура`, Роздріб: `1000 грн`, Великий опт: `850 грн`

- [ ] **Тест:** Індивідуальна ціна для VIP-клієнта
  - **Очікуваний результат:** Відображається персональна ціна
  - **Тестові дані:** Товар: `Монітор`, Індивідуальна ціна: `7500 грн`

- [ ] **Тест:** Акційні товари
  - **Очікуваний результат:** Відображається акційна ціна + бейдж "Акція"
  - **Тестові дані:** Товар: `Навушники`, Акційна ціна: `1200 грн` (замість `1500 грн`)

### 2.3. Швидке замовлення (Excel-like інтерфейс)
- [ ] **Тест:** Введення артикулу та кількості в таблицю
  - **Очікуваний результат:** Товар додано, ціна автоматично підставлена
  - **Тестові дані:**
    ```
    Артикул | Кількість
    SKU001  | 10
    SKU002  | 25
    SKU003  | 5
    ```

- [ ] **Тест:** Навігація клавіатурою (Tab, Enter, Arrow keys)
  - **Очікуваний результат:** Швидке переміщення між полями

- [ ] **Тест:** Автопідстановка при введенні артикулу
  - **Очікуваний результат:** Список підказок з товарами
  - **Тестові дані:** Введення `SKU0` показує всі артикули, що починаються з SKU0

- [ ] **Помилки для тестування:**
  - Введення неіснуючого артикулу
  - Введення кількості більше, ніж є на складі
  - Введення нульової кількості
  - Введення від'ємної кількості

### 2.4. Імпорт з CSV/Excel
- [ ] **Тест:** Імпорт CSV-файлу з замовленням
  - **Очікуваний результат:** Всі товари імпортовані, кошик заповнений
  - **Тестові дані:** Файл `order_template.csv`:
    ```csv
    sku,quantity
    SKU001,10
    SKU002,20
    SKU003,5
    ```

- [ ] **Тест:** Імпорт Excel-файлу (.xlsx)
  - **Очікуваний результат:** Товари імпортовані з правильними цінами
  - **Тестові дані:** Файл `order.xlsx` з стовпцями: Артикул, Кількість

- [ ] **Тест:** Імпорт з кириличними назвами
  - **Очікуваний результат:** Кодування UTF-8 правильно обробляється
  - **Тестові дані:** Файл з назвами товарів українською

- [ ] **Помилки для тестування:**
  - Файл з неправильною структурою стовпців
  - Файл з порожніми рядками
  - Файл > 1 МБ
  - Файл з > 1000 товарами (перевірка ліміту)
  - Файл з неіснуючими артикулами (частково валідні дані)

### 2.5. Копіювання з Excel (Ctrl+V)
- [ ] **Тест:** Скопіювати дані з Excel та вставити в інтерфейс
  - **Очікуваний результат:** Дані розпізнані, товари додані
  - **Тестові дані:** Скопіювати з Excel:
    ```
    SKU001    10
    SKU002    20
    SKU003    5
    ```

- [ ] **Тест:** Вставка даних з табуляцією
  - **Очікуваний результат:** Артикул та кількість правильно розділені

- [ ] **Тест:** Вставка даних з пробілами
  - **Очікуваний результат:** Пробіли як роздільники обробляються

- [ ] **Помилки для тестування:**
  - Вставка тексту без структури
  - Вставка > 500 рядків

### 2.6. Перевірка кредитного ліміту
- [ ] **Тест:** Замовлення в межах кредитного ліміту
  - **Очікуваний результат:** Замовлення підтверджено
  - **Тестові дані:** Кредитний ліміт: `100000 грн`, Замовлення: `50000 грн`

- [ ] **Тест:** Замовлення з перевищенням ліміту
  - **Очікуваний результат:** Попередження, пропозиція передоплати
  - **Тестові дані:** Кредитний ліміт: `100000 грн`, Замовлення: `150000 грн`, Залишок ліміту: `30000 грн`

- [ ] **Тест:** Відображення залишку кредитного ліміту
  - **Очікуваний результат:** В особистому кабінеті відображається актуальний залишок
  - **Тестові дані:** Ліміт: `100000 грн`, Використано: `60000 грн`, Залишок: `40000 грн`

- [ ] **Тест:** Поповнення після оплати
  - **Очікуваний результат:** Після підтвердження оплати ліміт відновлюється
  - **Тестові дані:** Оплата `30000 грн` збільшує доступний ліміт на `30000 грн`

### 2.7. Завантаження прайс-листа
- [ ] **Тест:** Завантаження прайсу в XLSX
  - **Очікуваний результат:** Файл завантажено, містить всі товари з актуальними цінами
  - **Перевірити:** Формат: Артикул, Назва, Ціна, Залишок, Опис

- [ ] **Тест:** Завантаження прайсу в CSV
  - **Очікуваний результат:** UTF-8 кодування, роздільник - кома
  - **Перевірити:** Кириличні символи відображаються правильно

- [ ] **Тест:** Завантаження прайсу в XML
  - **Очікуваний результат:** Валідний XML з правильною структурою
  - **Перевірити:** Схема XML відповідає стандарту YML (Yandex Market Language)

- [ ] **Тест:** Фільтрація прайсу за категорією
  - **Очікуваний результат:** Завантажуються тільки товари обраної категорії
  - **Тестові дані:** Категорія: "Ноутбуки"

- [ ] **Тест:** Персональні ціни в прайсі
  - **Очікуваний результат:** Відображаються ціни згідно рівня клієнта
  - **Перевірити:** VIP-клієнт бачить знижені ціни

### 2.8. Генерація фіду для Rozetka/Prom.ua
- [ ] **Тест:** Генерація фіду для Rozetka
  - **Очікуваний результат:** XML-файл у форматі Rozetka
  - **Перевірити:** Обов'язкові поля: id, name, price, currencyId, categoryId, picture, vendor

- [ ] **Тест:** Генерація фіду для Prom.ua
  - **Очікуваний результат:** XML-файл у форматі Prom.ua (YML)
  - **Перевірити:** Наявність полів: available, url, price, oldprice, vendor

- [ ] **Тест:** Фільтрація товарів (тільки в наявності)
  - **Очікуваний результат:** У фіді тільки товари зі статусом available="true"

- [ ] **Тест:** Оновлення фіду (cron job)
  - **Очікуваний результат:** Фід автоматично оновлюється раз на годину
  - **Перевірити:** Час останнього оновлення в метаданих файлу

- [ ] **Продуктивність:** Генерація фіду з 10000 товарами < 30 секунд

## 3. Dropshipping (Кабінет постачальника)

### Передумови
- Зареєстрований постачальник
- Налаштовані комісії та умови співпраці
- Підготовлені тестові файли для імпорту
- Налаштовані webhook для оновлень

### 3.1. Реєстрація постачальника
- [ ] **Тест:** Реєстрація нового постачальника
  - **Очікуваний результат:** Постачальник зареєстрований, отримано доступ до кабінету
  - **Тестові дані:**
    ```
    Назва компанії: ПП "Дропшип Постачальник"
    ЄДРПОУ: 87654321
    Email: supplier@dropship.com
    Телефон: +380671234567
    Категорія товарів: Електроніка
    ```

- [ ] **Тест:** Підписання договору (електронний підпис)
  - **Очікуваний результат:** Договір підписаний, статус "Активний"

- [ ] **Тест:** Налаштування комісії
  - **Очікуваний результат:** Комісія встановлена (стандартна 15%)
  - **Тестові дані:** Комісія: `15%`, Мінімальна маржа: `10%`

- [ ] **Помилки для тестування:**
  - Реєстрація з існуючим ЄДРПОУ
  - Реєстрація без банківських реквізитів

### 3.2. Додавання товару
- [ ] **Тест:** Додати новий товар вручну
  - **Очікуваний результат:** Товар доданий, статус "На модерації"
  - **Тестові дані:**
    ```
    Назва: Смартфон Samsung Galaxy A54
    SKU: SAMSUNG-A54-128-BLACK
    Ціна: 12000 грн
    Залишок: 50 шт
    Категорія: Смартфони
    Опис: 128GB, 8GB RAM, чорний колір
    ```

- [ ] **Тест:** Завантаження зображень (до 10 фото)
  - **Очікуваний результат:** Всі зображення завантажені, головне фото встановлено
  - **Тестові дані:** 5 фото в форматі JPG, кожне < 2 МБ

- [ ] **Тест:** Додавання характеристик товару
  - **Очікуваний результат:** Характеристики додані та відображаються
  - **Тестові дані:**
    ```
    Діагональ: 6.4"
    Процесор: Exynos 1380
    Оперативна пам'ять: 8 GB
    Вбудована пам'ять: 128 GB
    Камера: 50 MP
    Батарея: 5000 mAh
    ```

- [ ] **Тест:** Встановлення мінімальної ціни (floor price)
  - **Очікуваний результат:** Система не дозволить продати дешевше
  - **Тестові дані:** Мінімальна ціна: `11000 грн`

- [ ] **Помилки для тестування:**
  - Додавання товару без назви
  - Додавання товару без ціни
  - Завантаження фото неправильного формату (PNG > 5 МБ)
  - Дублікат SKU

### 3.3. Імпорт товарів (CSV/XML)
- [ ] **Тест:** Імпорт товарів з CSV
  - **Очікуваний результат:** Всі товари імпортовані

## 5. AI Репрайсинг

### Передумови
- Налаштоване підключення до API конкурентів

### 5.1. Створення правила репрайсингу
- [ ] **Тест:** Створити просте правило (beat_lowest)
  - **Очікуваний результат:** Правило збережено, застосовується до товарів
  - **Тестові дані:**
    ```
    Назва правила: Бити найнижчу ціну
    Стратегія: beat_lowest
    Різниця: -50 грн (наша ціна = найнижча - 50 грн)
    Категорія: Смартфони
    Мінімальна маржа: 10%
    Максимальна маржа: 30%
    ```

- [ ] **Тест:** Створити правило для конкретного товару
  - **Очікуваний результат:** Правило застосовується тільки до одного SKU
  - **Тестові дані:** SKU: `IPHONE-15-PRO`, Стратегія: `match_lowest`

- [ ] **Тест:** Створити правило з часовим обмеженням
  - **Очікуваний результат:** Правило активне тільки в певні години
  - **Тестові дані:** Активне: `09:00-18:00 Mon-Fri`

- [ ] **Тест:** Пріоритет правил (кілька правил на один товар)
  - **Очікуваний результат:** Застосовується правило з вищим пріоритетом
  - **Тестові дані:** Правило 1 (пріоритет: 10), Правило 2 (пріоритет: 5)

- [ ] **Помилки для тестування:**
  - Створення правила без назви
  - Створення правила з мінімальною маржею > максимальної
  - Створення правила без категорії та SKU (не вказано, до чого застосовувати)

### 5.2. Перевірка стратегій

#### 5.2.1. beat_lowest (Бити найнижчу ціну)
- [ ] **Тест:** Ціна конкурента: 10000 грн, різниця: -100 грн
  - **Очікуваний результат:** Наша ціна: 9900 грн
  - **Перевірити:** Маржа >= 10%

- [ ] **Тест:** Ціна конкурента дуже низька (менше мінімальної маржі)
  - **Очікуваний результат:** Наша ціна = мінімальна маржа (не опускаємося нижче)
  - **Тестові дані:** Собівартість: 9000 грн, Мін. маржа: 10%, Ціна конкурента: 9500 грн
  - **Очікувана ціна:** 9900 грн (9000 * 1.10), а не 9400 грн

#### 5.2.2. match_lowest (Співпадати з найнижчою ціною)
- [ ] **Тест:** Ціна конкурента: 12000 грн
  - **Очікуваний результат:** Наша ціна: 12000 грн

- [ ] **Тест:** Кілька конкурентів з різними цінами
  - **Очікуваний результат:** Наша ціна = найнижча серед конкурентів
  - **Тестові дані:** Конкурент 1: 12000 грн, Конкурент 2: 11500 грн, Конкурент 3: 12500 грн
  - **Очікувана ціна:** 11500 грн

#### 5.2.3. smart (Розумна стратегія)
- [ ] **Тест:** Розумна стратегія з урахуванням історії продажів
  - **Очікуваний результат:** Ціна оптимізована для максимального прибутку
  - **Тестові дані:**
    ```
    Історія продажів:
    - Ціна 10000 грн → 50 продажів/місяць
    - Ціна 9500 грн → 80 продажів/місяць
    - Ціна 9000 грн → 100 продажів/місяць
    ```
  - **Очікування:** AI обирає ціну, яка дає найбільший прибуток (не обов'язково найнижчу)

- [ ] **Тест:** Урахування еластичності попиту
  - **Очікуваний результат:** Для товарів з високою еластичністю ціна нижча

- [ ] **Тест:** Урахування залишків
  - **Очікуваний результат:** При великих залишках ціна знижується для швидкого продажу
  - **Тестові дані:** Залишок: 200 шт (при середньому продажу 20 шт/місяць)

- [ ] **Тест:** Урахування сезонності
  - **Очікуваний результат:** Ціна підвищується в сезон попиту
  - **Тестові дані:** Кондиціонери влітку, обігрівачі взимку

### 5.3. Обмеження ціни (мін/макс маржа)
- [ ] **Тест:** Мінімальна маржа 10%
  - **Очікуваний результат:** Ціна не опускається нижче собівартості + 10%
  - **Тестові дані:** Собівартість: 10000 грн, Мінімальна ціна: 11000 грн

- [ ] **Тест:** Максимальна маржа 30%
  - **Очікуваний результат:** Ціна не піднімається вище собівартості + 30%
  - **Тестові дані:** Собівартість: 10000 грн, Максимальна ціна: 13000 грн

- [ ] **Тест:** Абсолютні обмеження (floor/ceiling price)
  - **Очікуваний результат:** Ціна в межах встановлених лімітів
  - **Тестові дані:** Мінімальна ціна: 9500 грн, Максимальна ціна: 15000 грн

- [ ] **Тест:** Обмеження зміни ціни за раз (max 5%)
  - **Очікуваний результат:** Ціна змінюється не більше ніж на 5% за один репрайсинг
  - **Тестові дані:** Поточна ціна: 10000 грн, Рекомендована: 9000 грн
  - **Очікувана нова ціна:** 9500 грн (10000 * 0.95)

### 5.4. Ручний запуск репрайсингу
- [ ] **Тест:** Запустити репрайсинг для одного товару
  - **Очікуваний результат:** Ціна оновлена відповідно до правила
  - **Перевірити:** Час виконання < 5 секунд

- [ ] **Тест:** Запустити репрайсинг для категорії
  - **Очікуваний результат:** Ціни оновлені для всіх товарів категорії
  - **Тестові дані:** Категорія: "Смартфони" (100 товарів)
  - **Перевірити:** Час виконання < 30 секунд

- [ ] **Тест:** Запустити репрайсинг для всього каталогу
  - **Очікуваний результат:** Ціни оновлені для всіх товарів з правилами
  - **Тестові дані:** 1000 товарів
  - **Перевірити:** Час виконання < 5 хвилин

- [ ] **Тест:** Попередній перегляд змін (dry run)
  - **Очікуваний результат:** Показані зміни цін без застосування
  - **Перевірити:** Можливість схвалити або відхилити зміни

- [ ] **Помилки для тестування:**
  - Репрайсинг при відсутності даних конкурентів
  - Репрайсинг товару без собівартості

### 5.5. Історія змін цін
- [ ] **Тест:** Перегляд історії цін товару
  - **Очікуваний результат:** Графік зміни цін за період
  - **Тестові дані:** Період: останні 30 днів
  - **Перевірити:** Відображаються: дата, стара ціна, нова ціна, причина зміни

- [ ] **Тест:** Фільтрація історії за типом зміни
  - **Очікуваний результат:** Тільки автоматичні або тільки ручні зміни
  - **Тестові дані:** Фільтр: "Автоматичний репрайсинг"

- [ ] **Тест:** Експорт історії в Excel
  - **Очікуваний результат:** XLSX-файл з усіма змінами цін
  - **Перевірити:** Стовпці: SKU, Назва, Дата, Стара ціна, Нова ціна, Причина, Автор

- [ ] **Тест:** Порівняння з цінами конкурентів (графік)
  - **Очікуваний результат:** Графік показує нашу ціну та ціни 3 конкурентів
  - **Перевірити:** Легенда, різні кольори ліній

### 5.6. Сповіщення про ціни конкурентів

  - **Очікуваний результат:** Email/push-сповіщення
  - **Тестові дані:** Конкурент знизив ціну з 10000 грн до 9000 грн

- [ ] **Тест:** Сповіщення при появі нового конкурента
  - **Очікуваний результат:** Сповіщення про новий магазин з цим товаром

- [ ] **Тест:** Налаштування частоти сповіщень
  - **Очікуваний результат:** Збережені налаштування (instant, hourly, daily)

## 6. Прогнозування попиту

### Передумови
- Наявна історія продажів мінімум за 3 місяці
- Налаштовані параметри ML-моделі
- Імпортовані зовнішні фактори (свята, погода, тренди)
- Додані дані про постачальників та терміни доставки

### 6.1. Перегляд прогнозу для товару
- [ ] **Тест:** Прогноз попиту на наступні 30 днів
  - **Очікуваний результат:** Графік з прогнозом та довірчим інтервалом
  - **Тестові дані:** Товар: `iPhone 15 Pro`, Історія: 6 місяців
  - **Перевірити:** Верхня та нижня межі прогнозу (confidence interval 95%)

- [ ] **Тест:** Прогноз з урахуванням сезонності
  - **Очікуваний результат:** Прогноз показує підвищення попиту в сезон
  - **Тестові дані:** Товар: "Кондиціонер", Період: Червень-Серпень
  - **Перевірити:** Прогноз у червні > прогноз у грудні

- [ ] **Тест:** Прогноз з урахуванням тренду
  - **Очікуваний результат:** Прогноз показує загальний тренд (зростання/падіння)
  - **Тестові дані:** Товар з постійним зростанням продажів: +10% щомісяця

- [ ] **Тест:** Прогноз для нового товару (без історії)
  - **Очікуваний результат:** Прогноз на основі аналогічних товарів
  - **Тестові дані:** Новий товар: "iPhone 16", Аналог: "iPhone 15"

- [ ] **Тест:** Точність прогнозу (backtesting)
  - **Очікуваний результат:** Відображається MAPE (Mean Absolute Percentage Error)
  - **Перевірити:** MAPE < 20% (хороший прогноз)

- [ ] **Тест:** Прогноз на різні періоди (7, 30, 90 днів)
  - **Очікуваний результат:** Прогноз на кожен період з різною точністю
  - **Перевірити:** Чим довший період, тим ширший довірчий інтервал

### 6.2. Рекомендації закупівель
- [ ] **Тест:** Рекомендована кількість для закупівлі
  - **Очікуваний результат:** Кількість з урахуванням прогнозу, термінів доставки, страхового запасу
  - **Тестові дані:**
    ```
    Поточний залишок: 10 шт
    Прогноз на 30 днів: 50 шт
    Термін доставки: 7 днів
    Страховий запас: 20% (10 шт)
    ```
  - **Очікувана рекомендація:** Замовити 50 шт (прогноз) + 10 шт (страховий) - 10 шт (залишок) = 50 шт

- [ ] **Тест:** Рекомендована дата замовлення
  - **Очікуваний результат:** Дата з урахуванням терміну доставки
  - **Тестові дані:** Товар закінчиться через 15 днів, доставка 7 днів
  - **Очікувана дата:** Через 8 днів

- [ ] **Тест:** Оптимальне замовлення (EOQ - Economic Order Quantity)
  - **Очікуваний результат:** Кількість, що мінімізує загальні витрати
  - **Тестові дані:**
    ```
    Попит: 1000 шт/рік
    Вартість замовлення: 500 грн
    Вартість зберігання: 20% від вартості товару/рік
    Ціна товару: 1000 грн
    ```
  - **Перевірити:** Формула EOQ = sqrt((2 * попит * вартість_замовлення) / вартість_зберігання)

- [ ] **Тест:** Групування товарів для закупівлі (від одного постачальника)
  - **Очікуваний результат:** Список товарів для замовлення у постачальника
  - **Тестові дані:** Постачальник: "ТОВ Електроніка", 15 товарів для замовлення

- [ ] **Тест:** Пріоритизація товарів (ABC-аналіз)
  - **Очікуваний результат:** Товари розподілені на групи A, B, C
  - **Перевірити:**
    - Група A (20% товарів, 80% виручки) - висока увага
    - Група B (30% товарів, 15% виручки) - середня увага
    - Група C (50% товарів, 5% виручки) - низька увага

- [ ] **Тест:** Сповіщення про критичний рівень залишків
  - **Очікуваний результат:** Email/push-сповіщення
  - **Тестові дані:** Залишок 5 шт, мінімальний рівень 10 шт
  - **Перевірити:** Текст: "Критичний рівень залишків для [Товар]. Рекомендовано замовити 50 шт"

### 6.3. Аналіз сезонності
- [ ] **Тест:** Визначення сезонних патернів
  - **Очікуваний результат:** Графік показує піки продажів по місяцях
  - **Тестові дані:** Товар: "Купальники", Період: 2 роки
  - **Перевірити:** Піки у травні-серпні, спади у жовтні-березні

- [ ] **Тест:** Коефіцієнти сезонності по місяцях
  - **Очікуваний результат:** Таблиця з коефіцієнтами для кожного місяця
  - **Тестові дані:**
    ```
    Січень: 0.5 (50% від середнього)
    Червень: 1.8 (180% від середнього)
    Грудень: 1.2 (120% від середнього)
    ```

- [ ] **Тест:** Прогноз з урахуванням свят
  - **Очікуваний результат:** Підвищення попиту перед святами
  - **Тестові дані:** Товар: "Подарунки", Свята: Новий Рік, 8 Березня, День Валентина
  - **Перевірити:** Піки за 2-3 тижні до свят

- [ ] **Тест:** Вплив дня тижня на продажі
  - **Очікуваний результат:** Статистика продажів по днях тижня
  - **Перевірити:** П'ятниця та субота - найвищі продажі

- [ ] **Тест:** Експорт сезонного аналізу в Excel
  - **Очікуваний результат:** XLSX-файл з графіком та таблицею коефіцієнтів

### 6.4. Аналіз трендів
- [ ] **Тест:** Виявлення тренду зростання
  - **Очікуваний результат:** Лінія тренду з позитивним нахилом
  - **Тестові дані:** Продажі зростають на 5% щомісяця
  - **Перевірити:** Коефіцієнт тренду > 0

- [ ] **Тест:** Виявлення тренду падіння
  - **Очікуваний результат:** Лінія тренду з негативним нахилом
  - **Тестові дані:** Продажі падають на 3% щомісяця
  - **Перевірити:** Коефіцієнт тренду < 0, сповіщення про ризик

- [ ] **Тест:** Виявлення товарів-зірок (швидке зростання)
  - **Очікуваний результат:** Список товарів з найбільшим зростанням
  - **Тестові дані:** Фільтр: зростання > 20% за місяць
  - **Перевірити:** Рекомендація: "Збільшити закупівлі"

- [ ] **Тест:** Виявлення товарів, що застоюються
  - **Очікуваний результат:** Список товарів з падінням продажів
  - **Тестові дані:** Фільтр: падіння > 30% за 3 місяці
  - **Перевірити:** Рекомендація: "Розглянути зниження ціни або зняття з продажу"

- [ ] **Тест:** Кореляція з зовнішніми факторами
  - **Очікуваний результат:** Коефіцієнт кореляції з погодою, курсом валют, тощо
  - **Тестові дані:** Товар: "Морозиво", Фактор: "Температура"
  - **Перевірити:** Позитивна кореляція (чим вища температура, тим більше продажів)

- [ ] **Тест:** Автоматичне виявлення аномалій
  - **Очікуваний результат:** Позначені викиди на графіку
  - **Тестові дані:** Різкий стрибок продажів (у 3 рази вище норми)
  - **Перевірити:** Причина аномалії (акція, згадка в ЗМІ, тощо)

### 6.5. What-if симуляція
- [ ] **Тест:** Симуляція зміни ціни
  - **Очікуваний результат:** Прогноз попиту при новій ціні
  - **Тестові дані:**
    ```
    Поточна ціна: 1000 грн, попит: 100 шт/місяць
    Нова ціна: 900 грн
    Еластичність: -1.5 (при зниженні ціни на 10%, попит зростає на 15%)
    ```
  - **Очікуваний прогноз:** 115 шт/місяць

- [ ] **Тест:** Симуляція акції (знижка 20%)
  - **Очікуваний результат:** Прогноз попиту під час акції та після неї
  - **Тестові дані:** Акція 7 днів, знижка 20%
  - **Перевірити:** Підвищення попиту під час акції, можливе зниження після

- [ ] **Тест:** Симуляція виходу нового конкурента
  - **Очікуваний результат:** Прогноз зниження попиту
  - **Тестові дані:** Конкурент з ціною на 10% нижче
  - **Очікуване зниження:** 15-20% продажів

- [ ] **Тест:** Симуляція збільшення маркетингового бюджету
  - **Очікуваний результат:** Прогноз зростання попиту
  - **Тестові дані:** Збільшення бюджету на 50%
  - **Перевірити:** ROI (Return on Investment)

- [ ] **Тест:** Порівняння кількох сценаріїв
  - **Очікуваний результат:** Таблиця з порівнянням всіх сценаріїв
  - **Тестові дані:** 3 сценарії:
    1. Зниження ціни на 10%
    2. Акція зі знижкою 20% на 1 тиждень
    3. Збільшення маркетингового бюджету на 50%
  - **Перевірити:** Стовпці: Сценарій, Прогноз продажів, Виручка, Прибуток, ROI

- [ ] **Тест:** Експорт симуляції в PDF
  - **Очікуваний результат:** PDF-звіт з графіками та таблицями
  - **Перевірити:** Містить всі параметри симуляції та результати

## Загальні вказівки з тестування

### Продуктивність
- Час відгуку інтерфейсу < 200 мс
- Завантаження сторінки < 2 секунди
- Обробка великих файлів (> 1000 записів) < 60 секунд
- API запити < 500 мс

### Безпека
- Всі форми захищені від SQL-ін'єкцій
- Всі форми захищені від XSS-атак
- Автентифікація та авторизація працюють коректно
- Дані користувачів зашифровані (HTTPS)
- Логи не містять чутливої інформації (паролі, токени)

### Сумісність
- Тестування в браузерах: Chrome, Firefox, Safari, Edge
- Тестування на мобільних пристроях (iOS, Android)
- Тестування на різних роздільностях екрану
- Тестування з клавіатурою (accessibility)

### Зручність використання (UX)
- Зрозумілі повідомлення про помилки
- Підтвердження перед критичними діями (видалення, скасування)
- Індикатори прогресу для довгих операцій
- Можливість скасувати дію (undo)
- Автозбереження форм
- Підказки (tooltips) для складних полів

### Документація
- Кожна функція має опис у довідці
- Відеоінструкції для складних операцій
- FAQ з найчастішими питаннями
- Приклади використання (use cases)
